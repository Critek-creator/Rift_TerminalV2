# Verify lessons — Rift Terminal V2 — 2026-04-26

## tauri-webview-tauri-started-text-not-emitted-by-tauri-2
- **Polarity:** negative (profile bug)
- **Profile:** tauri-webview (FIRST live use 2026-04-26)
- **What happened:** The `tauri-webview.md` profile defines `tauri_started()` predicate as a literal text match for `"Tauri started"` (Tauri 2.x) or `"App started"` (1.x) in stderr. On a vanilla Tauri 2.10.3 setup (Rift's stack: Tauri 2 + Svelte 5 + xterm.js, no extra logging configured), neither text appears anywhere in stderr or stdout. The runtime spawns the WebView2 window silently. App was visibly healthy by every other observable signal — rift.exe alive at PID 20140, dev server returned HTTP 200 with proper Vite-served HTML, no panic markers — but the predicate scored fail because the search text is not a thing in vanilla Tauri 2.
- **The fix (profile-side options):**
  1. **Drop `tauri_started()` from default predicate set** for vanilla-Tauri-2 projects. Replace with composite `process_alive(after_seconds=N) AND dev_server_up(port=N)` which is empirically what "Tauri started" was meant to assert.
  2. **Add explicit Tauri-2 log line** — projects can opt-in by adding `tracing::info!("Tauri started")` (or eprintln!) at the top of their Tauri 2 setup hook, which makes the predicate match. Document this as the per-project enable in profile + sample src-tauri snippet.
  3. **Detect via window-title or process count** — alternative: predicate could check that a Tauri WebView2 child process exists alongside the binary. Tauri 2 spawns `msedgewebview2.exe` children; their presence is a real signal of "the WebView booted."
- **Recommendation:** Option (1) is the cleanest v1 fix — the composite already covers what users actually want to verify (the binary is up AND the dev server is up = the user-visible window is operational). Option (2) is opt-in for projects that genuinely want the explicit log marker. Option (3) is more invasive and platform-specific (WebView2 only on Windows; WKWebView on macOS doesn't spawn separate processes).
- **Applies to:** every Tauri 2 project using the tauri-webview profile until the profile is patched. AIDE (p003) will hit the same issue on its first verify run.
- **Source:** `/aegis --verify` first-run smoke-test on Rift Terminal V2, 2026-04-26 (HEAD ea96d9b)

## tauri-webview-process-tree-attribution
- **Polarity:** negative (coordinator gotcha)
- **Profile:** tauri-webview
- **What happened:** `desktop-commander start_process` returns the PID of the spawned shell (the parent of `npm run tauri:dev`), NOT the PID of the actual `rift.exe` Tauri binary. Naive use of `process_alive` on the returned PID checks the wrong process — the shell can be alive while rift.exe died, or vice versa. On Windows, the call chain is: `powershell.exe (PID 18604)` → `npm.cmd` → `tauri.cmd` → `cargo run` → `rift.exe (PID 20140)`. Coordinator must walk the process tree (or filter `Get-CimInstance Win32_Process` by name) to find the actual binary.
- **The fix:** Profile observation step should explicitly resolve "the Tauri binary PID" by name lookup (`Get-Process -Name $tauriBinName` on Windows, `pgrep -f` on Unix) using the binary name from `src-tauri/Cargo.toml [package].name` or `tauri.conf.json productName`. Capture both the parent shell PID (for teardown) AND the binary PID (for `process_alive` scoring). Document in profile observation section.
- **Applies to:** any Tauri profile that uses desktop-commander start_process for the spawn
- **Source:** `/aegis --verify` first-run smoke-test on Rift Terminal V2, 2026-04-26

## tauri-webview-teardown-no-cascade-windows
- **Polarity:** negative (profile bug — Windows-specific)
- **Profile:** tauri-webview
- **What happened:** Profile teardown spec says `mcp__desktop-commander__force_terminate(pid)` on the parent process should kill the spawn. On Windows, the call chain for `npm run tauri:dev` is: powershell → npm.cmd → tauri.cmd (Node.js) → cargo run → rift.exe → spawned msedgewebview2.exe children. Terminating the parent powershell does NOT cascade-kill grandchildren — they survive as orphans, holding port 1420 and consuming RAM until manually killed. Verified empirically: parent PID 18604 terminated cleanly per desktop-commander, but rift.exe (20140), 4 node.exe processes, and the port-1420 holder all remained alive.
- **The fix (profile-side):** Add explicit child-process cleanup to teardown. Per-platform recipes:
  - **Windows:** `Get-Process -Name "rift" | Stop-Process -Force; Get-NetTCPConnection -LocalPort <devPort> | Select-Object -ExpandProperty OwningProcess | Stop-Process -Force; Get-Process -Name "node" | Where-Object { $_.StartTime -gt $sessionStart } | Stop-Process -Force` (where `rift` is the binary name from `src-tauri/Cargo.toml [package].name`).
  - **Unix:** `pkill -P <parent_pid>` plus a fallback `lsof -ti tcp:<devPort> | xargs kill -9`.
  - **Generic:** profile should record `binary_name` + `dev_port` from project detection so teardown can build the kill list mechanically.
- **Verification step:** After teardown, MUST verify (a) binary process by name is gone, (b) dev port is free. The profile already mentions "Free port <devPort> — verify with curl returns connection refused" — but only "verify"; needs to be a hard gate that fails the teardown step if not actually free.
- **Recommendation:** Patch `tauri-webview.md` Teardown section with the Windows-specific recipe + binary-name resolution from Cargo.toml. AIDE (p003) and any other Tauri 2 project on Windows will hit this same teardown leak.
- **Applies to:** every Tauri 2 verify run on Windows (and possibly Unix too — Unix `pkill -P` only kills direct children, not grandchildren reliably either).
- **Source:** `/aegis --verify` first-run smoke-test on Rift Terminal V2, 2026-04-26 — observed orphans after force_terminate(parent_pid)
