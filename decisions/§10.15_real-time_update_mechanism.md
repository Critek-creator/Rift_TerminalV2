## TL;DR

Two-tier architecture:

| Tier | Direction | Mechanism | Why |
|------|-----------|-----------|-----|
| **1 — Internal** | Rust core → Svelte webview (in-process) | `tauri::ipc::Channel<T>` per stream + `app.emit*` for low-freq notifications + `app.emit_filter` for targeted window events | Tauri's `Channel<T>` is the only Tauri primitive designed for ordered, high-throughput streaming. The `event` system is documented as **"not designed for low latency or high throughput situations"** and is correct only for sparse signals. |
| **2 — External** | Translator processes ↔ Rift core (cross-process) | Tokio `broadcast` bus inside Rift core, fronted by a UDS (Unix) / named-pipe (Windows) IPC server with framed JSON envelopes + replay buffer | Mirrors V1's shipped Phase-7 IPC surface (proven on the `Rift_Terminal` repo). Required for the §9 translator-module pattern: Aegis, MCP servers, hook adapters all run as separate processes and must speak Rift's protocol over a process boundary. |

The two tiers meet in a single seam: a `rift-bus` crate that owns the tokio broadcast channel. In-process handlers subscribe directly; external translators connect via the IPC server, which is just another subscriber/publisher pair on top of the same bus.

---

## Throughput shape

What Rift actually has to move (rough orders of magnitude):

| Stream                    | Burst rate         | Steady-state | Latency budget |
|---------------------------|--------------------|--------------|----------------|
| PTY output (per tab)      | 10k+ lines/s       | low          | <16 ms (frame) |
| Filesystem events         | 100s/s during build| ~1/s idle    | <100 ms        |
| Hook events               | 1–5 per tool call  | bursty       | <100 ms        |
| Agent events              | 1–5/s              | low          | <250 ms        |
| Aegis events              | <1/s               | ~0           | <500 ms        |
| Status-line ticks         | 1/s                | 1/s          | <1 s           |

Tauri's `event` system handles the tail of this list trivially. The head — PTY output and FS-burst — needs `Channel<T>` to avoid serialization-per-event overhead and the event-system's intentional debounce.

## Tier 1 — Internal (Tauri-native)

### Pattern

```rust
// Rust side — one Channel per high-throughput stream
#[tauri::command]
fn pty_attach(app: AppHandle, tab_id: u32, stream: tauri::ipc::Channel<PtyChunk>) {
    let bus = app.state::<Mutex<RiftBus>>();
    bus.lock().unwrap().subscribe(SubscribeFilter::PtyTab(tab_id), move |evt| {
        let _ = stream.send(evt);  // backpressure: send returns Result
    });
}

// Low-frequency global signals stay on the event system
app.emit("session-saved", &session_id).unwrap();

// Window-targeted (cockpit detached window vs main)
app.emit_filter("graph-node-pinned", node_id, |target| match target {
    EventTarget::WebviewWindow { label } => label == "cockpit",
    _ => false,
}).unwrap();
```

### Frontend side

```ts
// Streams — Channel<T> for ordered high-throughput
import { invoke, Channel } from '@tauri-apps/api/core';
const ptyStream = new Channel<PtyChunk>();
ptyStream.onmessage = (chunk) => xterm.write(chunk.bytes);
await invoke('pty_attach', { tabId: 0, stream: ptyStream });

// Low-frequency — event listen
import { listen } from '@tauri-apps/api/event';
listen<string>('session-saved', (e) => statusline.flash('saved'));
```

### Decision matrix — Tauri internal

| Use                              | Mechanism            |
|----------------------------------|----------------------|
| PTY output per tab               | `Channel<PtyChunk>`  |
| Filesystem watcher fan-out       | `Channel<FsEvent>`   |
| Hook event log                   | `Channel<HookEvent>` (low-freq but ordered + bursty) |
| Agent event log                  | `Channel<AgentEvent>`|
| Config / theme reload            | `app.emit`           |
| Session save / load notifications| `app.emit`           |
| Cockpit-window-only signals      | `app.emit_filter`    |

## Tier 2 — External (translator processes ↔ Rift core)

### Why a process boundary exists

Per §9, translator modules can be:
- In-process Rust crates (built-in: `hook_translator`, `fs_translator`)
- **Out-of-process** (Aegis: private translator runs as a separate binary; MCP servers; Claude Code hook adapters launched by hooks themselves; any user-supplied integration written in any language)

Out-of-process is the load-bearing case. V1 already shipped this in Phase 7 (UDS + named pipe + framing + replay + the rift-tail consumer CLI as a reference client). Don't redesign — port the protocol shape, drop V1's specific code.

### Reused V1 architecture

- **Transport:** Unix domain socket (Linux/Mac) or named pipe (Windows). One socket per Rift instance.
- **Framing:** length-prefixed JSON envelopes (`{ version, category, type, payload, ... }`).
- **Bus:** tokio `broadcast` channel inside `rift-bus` crate, capacity-bounded; lagged subscribers get replay-buffer hits or DROP markers.
- **Replay:** pre-publish-before-start pattern (V1 lesson). New connections drain the replay buffer before live events stream — eliminates subscribe-vs-publish race.
- **Envelope versioning:** additive categories don't bump version (V1 lesson). Schema breaks bump.
- **Round-trip tests:** mandatory on every envelope variant (V1 lesson — bidirectional protocols are asymmetric trap).

### `rift-bus` crate shape

```rust
// crates/rift-bus/src/lib.rs (sketch)
pub struct RiftBus {
    tx: tokio::sync::broadcast::Sender<Envelope>,
    replay: ReplayBuffer,
}

pub enum SubscribeFilter {
    All,
    Category(Category),     // pty | hook | agent | fs | aegis | ...
    PtyTab(TabId),
    Window(WindowLabel),    // for cockpit-only events
    Custom(Box<dyn Fn(&Envelope) -> bool + Send + Sync>),
}

impl RiftBus {
    pub fn publish(&self, env: Envelope) -> Result<(), BusError>;
    pub fn subscribe(&self, filter: SubscribeFilter) -> Receiver;
    pub fn start_ipc_server(&self, addr: IpcAddr) -> Result<IpcHandle, IpcError>;
}
```

`start_ipc_server` is the surface external translators connect to. It's just another publisher/subscriber pair on the same bus — translators don't get a special path.

### Tier 2 decision matrix

| Situation                                                  | Mechanism                                       |
|------------------------------------------------------------|-------------------------------------------------|
| Built-in translator (e.g., `fs_translator`)                | In-process Rust, subscribes/publishes directly to `RiftBus` |
| Aegis translator (private, separate process)               | Connects to UDS/named-pipe IPC server, speaks Rift Integration Protocol |
| Claude Code hooks (themselves processes started by hooks)  | Connects to UDS/named-pipe IPC server, ephemeral |
| MCP servers                                                | Same — UDS/named-pipe                           |
| Future SSH-headless mobile client (post-v1)                | Same — bus already publishes everything; remote surface is just another subscriber |

## Backpressure, ordering, replay

| Property                | Tier 1 (Channel)               | Tier 2 (broadcast bus + IPC)                     |
|-------------------------|--------------------------------|--------------------------------------------------|
| Ordering                | Per-stream FIFO (Tauri)        | Per-receiver FIFO (tokio broadcast)              |
| Backpressure            | `send -> Result`; sender pauses| Bounded broadcast; lagged receivers get DROP markers + replay-buffer recovery |
| Replay (late subscriber)| N/A (point-to-point per stream)| Replay buffer drains on subscribe (V1 pattern)   |
| Multi-consumer          | One Channel per consumer       | tokio broadcast = native multi-consumer          |
| Crash recovery          | Channel closes; client reconnects via `invoke` | IPC client reconnects; replay buffer fills the gap |

---

## What this rules out

- ❌ **Tauri `event` system as the primary mechanism.** Documented as not for high throughput. Use it only for low-freq notifications.
- ❌ **WebSocket inside the webview as the primary mechanism.** Adds a network stack we don't need; serialization through JSON over WS is slower than Tauri's native `Channel<T>` for in-process. Reasonable as an *external* transport (mobile/SSH-headless post-v1) but not the foundation.
- ❌ **A custom IPC protocol from scratch.** V1's framing+replay layer is the right answer; reinventing it costs months.
- ❌ **One mechanism for everything.** The latency budgets diverge by 3 orders of magnitude across stream types — one-size-fits-all forces compromise.

---

## What it leaves open (for Phase 4 to nail down, not blockers)

- Exact replay buffer size (capacity vs memory tradeoff). Default to V1's value, tune with telemetry.
- Whether to expose `app.emit_filter` for cockpit-window-only events through `RiftBus`'s `SubscribeFilter::Window` or keep them on the raw Tauri event system. Slight preference for the former — single seam wins.
- The Rift Integration Protocol envelope schema is its own spec doc — write it during Phase 4 implementation, freeze when v1 ships per §9 spec discipline.

**New §10.15 body (vision v0.6):**

> ### 10.15 LOCKED — Real-time update mechanism (two-tier)
>
> **Tier 1 (in-process, Rust core → webview):** `tauri::ipc::Channel<T>` for high-throughput ordered streams (PTY output, filesystem events, hook/agent event log, cockpit graph updates); Tauri `app.emit`/`app.emit_filter` for low-frequency notifications and window-targeted signals. `Channel<T>` is non-negotiable for the streaming case — Tauri's event system is documented as not designed for high throughput.
>
> **Tier 2 (cross-process, translator modules ↔ Rift core):** UDS (Unix) / named pipe (Windows) IPC server inside Rift core, exposing a tokio `broadcast` bus through the **Rift Integration Protocol**. Length-prefixed JSON envelopes, replay buffer, additive-category versioning. This is V1's Phase-7-shipped pattern, ported in concept (not code).
>
> **Single seam:** a `rift-bus` crate owns the bus and the IPC server. In-process translators subscribe directly; external translators connect via the IPC server; both go through the same `SubscribeFilter` API. The webview-side `Channel<T>` handlers are also subscribers via Tauri command bridges.
>
> **Backpressure:** Tier 1 `Channel::send` returns Result; bus is bounded; lagged subscribers receive DROP markers and may use the replay buffer to recover.
> **Ordering:** per-stream FIFO (Tier 1 native, Tier 2 per-receiver via tokio broadcast).
> **Replay:** pre-publish-before-start (V1 lesson); new IPC connections drain the replay buffer before live events.
>
> Decision rationale: see `decisions/§10.15_real-time_update_mechanism.md`.
>
> Decision date: 2026-04-26. Reopen only on a concrete blocker.
