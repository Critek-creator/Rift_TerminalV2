## TL;DR

**Strategy D — Hybrid sentinel + injected boundaries + process-name fallback.**

| Layer | Mechanism | Lanes covered |
|------|-----------|---------------|
| **L1 — Shell-integration sentinels** | Inject `PROMPT_COMMAND` / `precmd` / `$prompt` / `fish_prompt` prelude at PTY spawn time. Prelude emits `\x1b]6973;<event>\x07` OSC sequences at known shell boundaries (prompt-start, prompt-end, command-start, command-end, exit-status). Parser strips sentinels from the byte stream and tags lanes. | `SYS` (prompt), user-input echo, `OK` / `WARN` / `ERR` (from `$?` / `$LASTEXITCODE`) |
| **L2 — Bus-translator-injected sentinels** | The hook translator (which holds `PtyControl`) writes `\x1b]6973;HOOK_START\x07` / `_END\x07` directly into the PTY input stream at hook boundaries. Sentinels arrive in the byte stream in **causal order** with the hook's own output — no clock comparison, no race. | `HOOK`, `AEGIS` |
| **L3 — Process-name fallback** | Best-effort: if no sentinel marks a region, query the foreground child PID via `child.process_id()` (portable-pty exposes this) and look up the process name. `claude` / `node` (when running CC) → `CLAUDE` lane. Graceful degradation: unknown process → no lane tag (passes through). | `CLAUDE` (best-effort, upgrades to sentinel-based when CC config emits sentinels) |

**Why D and not the alternatives:**

- **Strategy A (process-detection alone)** is structurally invalid on Windows — ConPTY exposes no foreground-PID API to an external observer (`GetConsoleProcessList` requires `AttachConsole`, conflicts with Rift's own console). Also can't disambiguate hook lines from CC lines when they share a process.
- **Strategy B (hook-spool time-correlation)** is non-compliant with §10.1 baseline — without integrations loaded, zero envelopes flow → can't distinguish OK / ERR / SYS at all on bare-install. xterm.js has no "go back and recolor line N" API; retroactive classification is structurally impossible. Also racy under sustained throughput (envelope arrives 15–50 ms after the bytes that produced it).
- **Strategy C alone** covers shell prompt boundaries cleanly but leaves CLAUDE / HOOK / AEGIS lanes unaddressed without active CC + Aegis cooperation.

**D solves B's race by replacing time-correlation with stream injection.** The hook translator writes the sentinel into the PTY stream synchronously with the hook event, so the sentinel arrives *in causal order* relative to the hook's own output. No clock comparison, no window heuristics — the byte stream itself carries the boundary.

## Sentinel format

### OUI choice — `6973`

OSC sequences use `\x1b]<NUM>;<DATA>\x07` (or `\x1b]<NUM>;<DATA>\x1b\\`). The `<NUM>` is the OSC command code. Rift uses `6973` as a Rift-specific OUI:

- Not in any registered OSC range (registered codes: 0/1/2 title, 4 color-set, 8 hyperlink, 10/11/12 fg/bg/cursor, 52 clipboard, 133 shell-integration, etc.).
- Far enough from common values that legitimate program output is unlikely to collide.
- Two ASCII characters (`6973`) — easy to grep for in logs.

Pattern: `\x1b]6973;<event>[;<key>=<value>]*\x07`

Allowed events:
| Event | Emitted by | Marks |
|-------|-----------|-------|
| `PROMPT_START` | shell prelude (L1) | Open `SYS` lane (prompt region) |
| `PROMPT_END`   | shell prelude (L1) | Close `SYS`, await user input |
| `CMD_START`    | shell prelude (L1) | Close user-input region, await command output |
| `CMD_END;exit=N` | shell prelude (L1) | Close command-output region, lane = `OK`/`WARN`/`ERR` based on N |
| `HOOK_START;name=<hook>` | hook translator (L2) | Open `HOOK` lane |
| `HOOK_END;name=<hook>` | hook translator (L2) | Close `HOOK` lane, restore previous |
| `AEGIS_START` / `AEGIS_END` | Aegis translator (L2) | Open / close `AEGIS` lane |
| `CLAUDE_START` / `CLAUDE_END` | (optional, opt-in) `RIFT_CLAUDE_SENTINELS=1` + user CC config | Open / close `CLAUDE` lane |

### Lane state machine

The parser maintains a per-session **lane stack** (Vec<Lane>). Default lane = `SYS` for prompt regions; default for command-output regions = `OK` (until CMD_END indicates otherwise) or process-name-derived.

- `*_START` push; `*_END` pop. Stack-based so nested regions (e.g., a hook fires while CC is responding) don't corrupt state.
- Stack max depth: 8. Overflow → drop oldest, log warning.
- `PROMPT_START` clears the stack and pushes `SYS` — defensive recovery if a previous region didn't close.

### Per-shell prelude

Set via `PtyOptions::env`-injected `RIFT_SHELL_PRELUDE` env var. Rift's `pty_start` Tauri command resolves the user's `ShellPref` and passes the matching prelude before spawn:

| Shell | Mechanism | Inject point |
|-------|-----------|--------------|
| `bash` / `zsh` | `PROMPT_COMMAND` (bash) / `precmd` + `preexec` (zsh) | Append to existing — preserve user dotfile additions |
| `pwsh` | Override `$Function:prompt` | Wrap user's existing `prompt` function if present |
| `fish` | `fish_prompt` / `fish_preexec` / `fish_postexec` events | Add Rift hook via `function ... --on-event ...` |
| `nu` | `$env.PROMPT_COMMAND` closure | Wrap user closure |
| `cmd.exe` | **Dead zone** — no `PROMPT_COMMAND` equivalent | Inject `PROMPT $E]6973;PROMPT_START$E\\$P$G` (limited; only PROMPT_START fires; `CMD_END` impossible without ENABLE_VIRTUAL_TERMINAL_INPUT macro). Documented degradation: `cmd.exe` falls back to single-lane mode with a one-time toast. |
| `sh` (busybox) | No `PROMPT_COMMAND` reliably | Documented degradation as `cmd.exe`. |

Prelude scripts live at `crates/rift-bus/src/translators/lane_prelude/<shell>.{sh,ps1,fish,nu}` — embedded into the binary via `include_str!`. Shipping them as files in `~/.config/rift/` is rejected — too much install-time surface.

## Architecture

### Translator boundary (§9 placement)

```
┌────────────────────────────────────────────────────────────┐
│ src-tauri/src/lib.rs (pty_start command — Rift host)       │
│  ─ resolves ShellPref → (path, args)                        │
│  ─ asks lane_translator for prelude → injects RIFT_SHELL_PRELUDE
│  ─ spawns PtySession via crates/rift-core/src/pty.rs        │
└────────────────────────────────────────────────────────────┘
                          │
                          ▼  (PTY bytes, raw, via Channel<T>)
┌────────────────────────────────────────────────────────────┐
│ crates/rift-bus/src/translators/lane.rs (NEW — this decision) │
│  ─ wraps PtyOutput::recv with sentinel-stripping parser     │
│  ─ maintains per-session lane stack                          │
│  ─ emits (Vec<u8>, Option<Lane>) tuples                      │
│  ─ subscribes to Category::Hook + Category::Aegis bus events │
│    to RECEIVE INSTRUCTIONS for sentinel injection            │
│  ─ holds PtyControl reference per-session for inject calls  │
└────────────────────────────────────────────────────────────┘
                          │
                          ▼  (Vec<u8> bytes-with-lane, ANSI-prefixed)
┌────────────────────────────────────────────────────────────┐
│ src/lib/Terminal.svelte (xterm.js consumer)                 │
│  ─ term.write(bytes) — bytes already carry lane ANSI prefix │
│  ─ no further work; rendering is per existing pipeline      │
└────────────────────────────────────────────────────────────┘
```

`lane.rs` lives inside `crates/rift-bus/src/translators/` per §9. It's a **transformer** translator (input = PTY bytes, output = lane-tagged PTY bytes) — different shape from the existing emitter translators (`status.rs`, `vault_walker.rs`) which read external sources and *publish* envelopes. The `commands.rs` translator is the closest precedent (it parses PTY bytes and emits structured events).

### `lane.rs` — module shape

```rust
//! Translator: live PTY-stream lane classification per §10.1.
//!
//! Wraps `PtyOutput` from rift-core; strips `\x1b]6973;…\x07` sentinels;
//! emits ANSI-prefixed bytes that match the §10.1 palette. Per-session
//! state is task-local (one `LaneState` per spawned `pty_start`).

pub fn shell_prelude(pref: &ShellPref) -> Option<&'static str> { … }
pub fn make_lane_translator() -> LaneTranslator { … }

pub struct LaneTranslator {
    state: LaneState,         // per-session; lane stack, partial-sentinel buffer
    bus: RiftBus,              // for receiving HOOK/AEGIS bus events
    pty_control: PtyControl,   // for injecting sentinels via PtyControl::write
}

impl LaneTranslator {
    /// Drive one chunk through the parser. Returns the (potentially modified)
    /// chunk with lane ANSI escapes inserted at lane boundaries.
    pub fn process(&mut self, chunk: &[u8]) -> Vec<u8> { … }

    /// Inject a hook-start sentinel into the PTY's input stream. Called by
    /// the hook translator when a hook is about to fire.
    pub fn inject_hook_start(&self, name: &str) { … }
    pub fn inject_hook_end(&self, name: &str) { … }
}
```

### Hook translator coordination

The hook translator (already shipped) currently publishes `Category::Hook / kind=hook.{started,finished}` envelopes. To wire into the lane system, it needs a reference to the active PTY's `LaneTranslator` so it can call `inject_hook_start(name)` *before* invoking the hook subprocess and `inject_hook_end(name)` *after* it returns.

This requires plumbing `LaneTranslator` access into the hook-fire path. Options:
- (a) Hook translator holds a `Weak<LaneTranslator>` per session; lookup by session-id when firing.
- (b) Hook translator publishes a `Category::System / kind=lane.inject` envelope; lane translator subscribes and self-injects.

**Decision: (a).** Direct injection. (b) reintroduces a cross-task hop and a tiny race (lane envelope vs hook output). The bus is for *observation* of state; the lane sentinels need *causal* placement in the PTY stream. Direct call is clean.

### Frontend

No frontend changes. `lane.rs` outputs ANSI-prefixed bytes that xterm.js renders natively (24-bit color via `\x1b[38;2;R;G;Bm` matches what `laneFormat.ts` already emits for Rift-OWNED lines). Tag prefix style on the terminal surface uses bracketed ASCII (`[CLAUDE] message`) rather than the bordered-box style used in HTML lanes — monospace cells can't render the bordered-box style cleanly. This matches the existing `laneFormat.ts` shape.

`RiftConfig.terminal.lanes_enabled` (already shipped) gates the entire pipeline:
- `false` → `lane.rs` is bypassed; `pty_start` doesn't inject `RIFT_SHELL_PRELUDE`; bytes flow through unmodified.
- `true` (default) → full pipeline active.

## Critical Unknowns — gate prototype

### CU-1 — Foreground-process API on Windows ConPTY (BLOCKING for L3 only)

**Question:** Can Rift query the current foreground PID of a ConPTY-spawned shell from outside the PTY session?

**Why it matters:** L3 (process-name CLAUDE fallback) needs `child.process_id()` for the *root* child plus a way to walk the process tree to find the current foreground. `child.process_id()` gives root only; finding "current foreground" requires platform-specific code.

**Test:**
```
1. Spawn ConPTY shell via portable-pty.
2. From the shell, run `claude --version`.
3. While `claude` is running, attempt to detect it from the Rift process via:
   a. Walking child processes of root PID (Windows: `CreateToolhelp32Snapshot` + `Process32First/Next`).
   b. `NtQueryInformationProcess` with `ProcessBasicInformation`.
4. Measure: success rate, latency, false-positive rate during process transitions.
```

**Expected:** Process-tree walking via `CreateToolhelp32Snapshot` works but is slow (~5–20 ms per scan) and racy. **Acceptable for L3 fallback only** because L3 is not on the per-chunk hot path — sample at command-start (CMD_START sentinel) and cache the lane decision until CMD_END.

**Blocking verdict:** Failure of CU-1 means L3 degrades to "no CLAUDE-lane fallback on Windows; users opt in via `RIFT_CLAUDE_SENTINELS=1` and CC config." Acceptable.

### CU-2 — Per-chunk classification latency budget (BLOCKING)

**Question:** What is p99 time between `output.recv().await` returns under sustained bursty output?

**Why it matters:** The classification work runs synchronously inside the drain task. If it takes >1 ms per chunk under load, the mpsc channel fills, applying backpressure to the std::thread reader, which blocks on `send()`. PTY master read buffer fills, child write syscalls stall, paging stalls.

**Test:**
```
1. Spawn `cat /dev/zero | head -c 100MB` through PTY (Linux).
2. Time delta between consecutive `output.recv().await` returns; record p50/p95/p99/max.
3. Repeat with the lane parser wired in. Compare deltas.
4. Repeat for `head -c 1G` to find sustained-throughput failure mode.
```

**Acceptance threshold:** Lane parser overhead must keep p99 < 1 ms per 4 KiB chunk. Sentinel scanning is a single linear pass per chunk (~5 µs in microbenchmark), well under threshold. The risk is the carry-over buffer for split sentinels — must be bounded (≤64 bytes).

**Blocking verdict:** Failure of CU-2 means the parser must move to a separate task with its own channel (extra hop, extra allocation). Prefer-pass state.

### CU-3 — ConPTY OSC sequence fidelity on Windows (BLOCKING)

**Question:** Does ConPTY on Windows 10 / 11 transmit `\x1b]6973;…\x07` byte-for-byte through the master fd, or does it intercept / strip / transform OSC sequences?

**Why it matters:** Strategy D's L1 + L2 layers both rely on OSC sentinels surviving the round-trip from shell → ConPTY → reader thread. Some older Windows 10 builds (pre-1903) had ConPTY versions that stripped OSC sequences they didn't recognize.

**Test:**
```
1. Spawn pwsh via ConPTY.
2. From pwsh: `[Console]::Write("`e]6973;TEST`a")`  # writes \x1b]6973;TEST\x07
3. Read from PTY master; assert byte sequence appears verbatim.
4. Repeat for cmd.exe, bash via WSL.
5. Repeat across Windows 10 22H2 + Windows 11 23H2 + Windows Server 2022.
```

**Acceptance threshold:** Must round-trip verbatim on Windows 11 23H2 (latest stable). Older Windows 10 builds may fail; documented as "Windows 10 lane classification requires 22H2+."

**Blocking verdict:** Failure of CU-3 on Windows 11 = Strategy D fundamentally unviable on Windows; pivot to L3-only (process-tree walk) Windows degradation. Failure only on older Win10 = acceptable; document the floor.

## Failure modes (with explicit mitigations)

| Failure | Likelihood | Mitigation |
|---------|-----------|-----------|
| **Sentinel forgery** — user runs `cat evil.sh` containing `\x1b]6973;HOOK_START\x07` | Low (OUI 6973 unique to Rift) | Forged region closes at next `PROMPT_START` (always emitted by shell). Blast radius: one inter-prompt interval. Acceptable — forging is harmless beyond visual confusion. |
| **Starship / oh-my-zsh overwrites `PROMPT_COMMAND`** | High (popular tools) | Prelude APPENDS to existing `PROMPT_COMMAND` (semicolon-joined for bash). Tested matrix: bare bash, oh-my-zsh, Starship, posh-git. Documented as a smoke-test before shipping. |
| **Alt-screen TUI app (vim, less, htop)** — no prompt sentinels fire for entire app duration | Guaranteed (every TUI app does this) | Detect `\x1b[?1049h` (alt-screen enter) → push `SYS` lane, pause sentinel parsing for body. On `\x1b[?1049l` (alt-screen exit) → pop, resume. Bytes inside alt-screen pass through with `SYS` tag (correct — it's user-requested terminal content). |
| **`cmd.exe` no PROMPT_COMMAND** | Guaranteed when user picks cmd | Documented dead zone. One-time toast on first cmd.exe spawn: "Lane classification limited under cmd.exe — use pwsh for full coverage." |
| **Parser desync from interleaved OSC** (e.g. tmux nesting) | Medium | State-machine parser handles nested `\x1b]…\x07` correctly. On parse error: drop current lane stack, reset to `SYS`, log warning. Recovery point: next `PROMPT_START`. |
| **Cross-chunk sentinel split** (sentinel lands halfway across two 4 KiB chunks) | Common (sentinels are 16–64 bytes; chunks are 4096) | Parser holds a 128-byte carry-over buffer. If buffer non-empty at chunk-end, prepend to next chunk. Bounded. |
| **`pty_resize` mid-classification** | Common | Lane state survives resize (no state in pixel/cell coords). Bytes emitted by terminal in response to SIGWINCH are passed through with current lane. |
| **`pty_kill` + new session** | Common | Per-session `LaneState` is task-local; aborted with the drain task. New session starts with fresh state. |
| **Webview reload mid-session** | Rare | `LaneState` is in Rust drain task — survives webview reload. First post-reload bytes carry the last-known lane (correct continuity). |
| **Multiple concurrent PTY sessions** | Common | One `LaneTranslator` per session; per-session state. No shared state. |

## Implementation plan (post-prototype)

### Phase 1 — Prototype on feature branch (gates: CU-2, CU-3 measurement)

- Branch `feat/d-018-lane-classification`.
- Skeleton `crates/rift-bus/src/translators/lane.rs`: state machine + sentinel parser, no shell prelude.
- Bench `cat /dev/zero | head -c 1G` through the parser. Record p50/p95/p99 chunk-to-chunk delta. Close CU-2.
- Run on Windows 11 23H2 + Windows 10 22H2: verify OSC `\x1b]6973;…\x07` survives ConPTY round-trip. Close CU-3.
- **Decision gate.** If CU-2 fails → restructure as separate task. If CU-3 fails → pivot to L3-only on Windows.

### Phase 2 — Shell prelude matrix (gates: shell coverage tests)

- Embed `lane_prelude/{bash,zsh,pwsh,fish,nu}.{sh,ps1,fish,nu}` via `include_str!`.
- Plumb `RIFT_SHELL_PRELUDE` env var injection through `pty_start` (`src-tauri/src/lib.rs:pty_start`) when `RiftConfig.terminal.lanes_enabled = true`.
- Test matrix: 5 shells × {bare, with-Starship, with-oh-my-zsh/posh-git} = 15 spawn scenarios. Each must produce expected sentinel sequence on `ls; ls /no-such; <Ctrl+C>`.

### Phase 3 — L2 hook integration

- Hook translator gains `Weak<LaneTranslator>` per session. On hook-fire: `inject_hook_start(name)` before subprocess invocation; `inject_hook_end(name)` after.
- Same wiring for Aegis translator (when `aegis` feature is on).

### Phase 4 — L3 process-name CLAUDE fallback

- On `CMD_START` sentinel: if no `CLAUDE_START` opt-in present in the spawned shell, walk child processes of `pty_root_pid` (using platform-specific code), find the foreground, look up name. If `claude` / `node` / known CC binary → push `CLAUDE` lane on the stack. Pop on `CMD_END`.
- Sampled once per command (not per chunk) — no hot-path cost.
- Linux: `/proc/<pid>/comm`. macOS: `proc_pidpath`. Windows: `CreateToolhelp32Snapshot` + `Process32First/Next` to find first child of `pty_root_pid`, then `GetModuleBaseName`.

### Phase 5 — Ship

- Lock the `\x1b]6973;…` sentinel format (post-ship breaking changes require schema-version bump).
- Translator-boundary check exempts `lane.rs` (it's already inside `translators/`).

## Locked decisions (non-negotiable post-decision)

1. **Strategy D — hybrid sentinel + bus-injected boundaries + process-name fallback.** Strategies A, B, C alone are rejected.
2. **OSC OUI = `6973`.** Format `\x1b]6973;<event>[;<key>=<value>]*\x07`.
3. **L1 events fixed list:** `PROMPT_START` / `PROMPT_END` / `CMD_START` / `CMD_END;exit=N`. New events require a doc revision.
4. **L2 events fixed list:** `HOOK_START;name=<name>` / `HOOK_END;name=<name>` / `AEGIS_START` / `AEGIS_END` / `CLAUDE_START` / `CLAUDE_END`.
5. **Direct `Weak<LaneTranslator>` reference** for hook translator (option (a)). Bus-event coordination (option (b)) is rejected — race-prone.
6. **`cmd.exe` is a documented dead zone.** Single-lane fallback with one-time user toast.
7. **Bordered tag prefix on terminal surface uses bracketed ASCII** (`[CLAUDE] message`), not bordered-box (HTML lanes only). Matches existing `laneFormat.ts`.
8. **Per-session task-local `LaneState`.** No shared state across sessions.
9. **`RiftConfig.terminal.lanes_enabled = false` bypasses the entire pipeline** — no prelude injection, no parser. Default `true`.
10. **Prototype gates on CU-2 + CU-3 measurement** — code does not land on `main` until both close green.
