Defer further surface work to operational experience. Building a grouping UI before users ask for it is the §7 anti-shortcut version of "feature-creep guised as completeness."

## What's being grouped

Agent.* envelopes shipped in Phase 5.x (hook-derived adapter):

| Kind              | Payload fields                                       |
|-------------------|------------------------------------------------------|
| `agent.dispatch`  | `{ name: String, input_digest: String }`             |
| `agent.complete`  | `{ name: String, success: bool, duration_ms: u64 }`  |
| `agent.error`     | `{ name: String, error_class: String, details: String }` |

**`name`** is the value passed as `subagent_type` to the Agent tool. Today (Phase 7 Tranche-3 ship) the population is small — handful of personal agents + a few plugin agents. As the catalog grows it becomes a natural grouping key.

**Implicit grouping primitive that already exists**: lifecycle pairing. A `dispatch` plus its later `complete` (or `error`) are a single logical "run". The hook-adapter doesn't surface a dispatch_id today (input_digest is approximate but not stable across the pair). v1.2's "group by run" would require either:
- adapter-side change to emit a stable dispatch_id and propagate it through complete/error, OR
- frontend-side correlation heuristic on `(name, input_digest)` matching the most-recent unmatched dispatch within a time window.

The adapter-side change is the right answer if v1.2 lands; the heuristic is fragile.

## What users would actually want to filter

Ranked by likely value (informed estimate, no operational signal yet):

1. **Status: "show me what's still running"** — single highest signal. After dispatching multiple parallel subagents, "which of these have finished" is the natural question. Pairs `dispatch` events without matching `complete`/`error` against a 5-min staleness clock; renders them with a pulsing amber badge.

2. **Errors only** — "did anything fail in the last hour?" filter. Already trivially derivable from `agent.error` envelope subscription; no new schema needed.

3. **By agent name** — once the personal+plugin catalog crosses ~10 agents, "show only rust-expert dispatches" becomes useful for retrospective debugging ("when did we last fire that?").

4. **By duration bucket** — "show only dispatches that took >60s" — useful for cost/perf retrospectives. Requires `duration_ms` from `complete`; trivial filter once the lifecycle correlation from #1 exists.

5. **By dispatcher (post-Sentinel)** — "show only Aegis-orchestrated agents" vs "show only Sentinel-flagged misbehavior". Distinct sources need distinct visual lanes; Sentinel's misbehavior events are §10.11 territory and should NEVER mix with the routine agent.* lifecycle stream. v1.0 doesn't have this concern (Sentinel is a placeholder card per Phase 7.5); becomes relevant only when Sentinel ships its own envelope category (sentinel.* — additive per envelope-version-additive-categories-no-bump rule).

#1 + #2 are 80% of the value. #3-5 are 80% of the implementation surface. Stage gating accordingly.

## v1.1 — three sub-tabs (proposed when needed)

Inside the AegisLog tab's tab-anatomy header (§10.8), add a thin sub-tab bar:

```
[ All  Running  Errors ]   ← faint amber buttons, active = filled
─────────────────────────
... existing 4 sections render below, filtered by sub-tab selection ...
```

- **All**: existing flat chronological list (current behavior).
- **Running**: dispatches without matching complete/error within last 5 minutes; staleness badge after 5 min.
- **Errors**: agent.error envelopes from session start (or last N=50, whichever is fewer).

Sub-tab state: per-tab, NOT persisted across sessions. Resets to "All" on app launch — keeps the lazy default predictable.

Implementation hint (when v1.1 lands): the existing `bus_subscribe('agent', ...)` handler maintains a `$state` array of envelopes; the sub-tab toggle drives a `$derived` filter over that array. No backend change. ~80 LOC frontend.

## v1.2 — group-by-name dropdown (proposed if v1.1 isn't enough)

Header dropdown: `Group by: [time ▾]` with options `[time, agent name]`.

- **time** (default): existing chronological list, no headers between events.
- **agent name**: collapsible group headers per unique `name` value, sorted by recency of most-recent envelope in each group. Each group has its own count badge ("rust-expert · 7 events"). Click header to collapse / expand the group.

Group-by state: persisted to `rift-config.toml` under a new `[cockpit.agent_log] group_by` key. Survives session restarts. Default `"time"` on first launch.

Implementation hint: this is heavier than v1.1 because it interacts with the live activity strip (§10.8 4-section anatomy) — the strip currently shows latest-first; if grouped by name, the strip should still show globally-most-recent across all groups, not per-group. ~200 LOC frontend + config schema bump (additive — `group_by` defaults to `"time"` so older configs don't break).

## What v1 explicitly does NOT do

- **No dispatch_id propagation** in the hook-adapter. Lifecycle pairing in v1.1 uses the (name, input_digest, time-window) heuristic; if it proves fragile, the adapter-side change becomes a separate beat.
- **No filter persistence in v1.1.** Sub-tab state is per-session-and-per-tab.
- **No regex / fuzzy filtering** in either v1.1 or v1.2. The v1.2 dropdown is enum-only.
- **No saved filter presets.** If users want "show me the last hour of sonnet dispatches that took >30s and errored," that's a v2 query-builder surface — out of scope.
- **No Sentinel mixing.** Sentinel events (when D-010 closes post-v1) get their own section card / tab routing; they NEVER appear in the AegisLog filter set.

## Why v1.0 ships flat

Three reasons:

1. **The catalog is still small.** With ~6 personal agents + a handful of plugin agents in active use, scrolling through a session's worth of `agent.*` events doesn't hit the friction threshold that justifies grouping UI. Building grouping for a 12-event-per-session world would be the §7 "no shortcuts on load-bearing visuals" rule applied in reverse — over-investing in chrome before it carries weight.

2. **The hook-adapter's input_digest field is opaque to v1.** Today it's "Input digest pending β" (per p006 5.3a notes). Without a stable correlation key, lifecycle-pairing heuristics in v1.1/v1.2 are best-effort. Better to ship the adapter improvement first IF v1.0's flat list proves insufficient — let operational signal drive the right adapter change.

3. **Sentinel reshapes the picture.** §10.11's Sentinel-Rift split says agent-misbehavior detection is Sentinel's job, not Rift's. When Sentinel ships its own envelope category, the AgentLog tab grows a peer view (Sentinel section / tab) for misbehavior events. Building elaborate grouping in the AegisLog tab now would partially duplicate work that Sentinel will reframe later. v1.0 holds the surface flat; v2+ rebuilds the right grouping with both event sources.

The recommendation pattern here mirrors §10.15's two-tier decision: pick the smallest correct surface for v1, document the staged path forward, and let operational experience drive the next beat.
