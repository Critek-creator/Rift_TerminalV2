# §10.18 — Graph Library Decision (Phase 8 Index Graph View)

*Decided: 2026-04-27 by `/aegis --research` (RESEARCH MODE, TIER-TEAM, 3 parallel haiku scouts on Context7 docs)*
*Inputs: Cytoscape.js v3.33.2 (released 2026-04-06), D3 v7, Sigma.js v3 (latest 2024); §10.3 visual aesthetic (matte black + amber/terminal lanes + JetBrains Mono + scanlines + CRT vignette, non-negotiable); §10.12 Index-as-tab+graph two-views-same-data; §10.14 Index data-enrichment layer; §10.15 real-time update mechanism (Tauri Channel<T> Tier 1); §11 GUI Cockpit vision (drag-into-terminal, click-to-pin, glow-on-touch); §9 Integration Decoupling.*
*Status: RESOLVED — D3 (modular: `d3-force` + `d3-selection` + `d3-zoom` + `d3-drag`) selected. Phase 8 unblocked; code starts at Phase 8 entry.*

---

## TL;DR

**Phase 8 ships D3 (v7, modular import) for the Index graph surface.** Three candidates evaluated against Rift's locked aesthetic, drag-into-terminal requirement, real-time update fit, and vault-scale perf headroom:

| Lib        | Aesthetic match | Bundle (gzip) | Perf at 200 nodes | Perf at 1000+ nodes | Idiom fit (§9) |
|------------|-----------------|---------------|-------------------|---------------------|----------------|
| **D3** ✅   | HIGH (SVG filters trivial)        | ~50 KB modular | 60 FPS SVG | cliff (Canvas escape hatch) | primitives-first — best |
| Cytoscape  | medium (CSS-tricks for glow)      | heavy (~400 KB suspected) | 60 FPS Canvas | 60 FPS WebGL | opinionated lib worldview |
| Sigma      | medium-high (custom GLSL ~3h)     | ~150-200 KB    | 60 FPS WebGL | 60 FPS WebGL | opinionated, no Canvas fallback |

D3 wins on the two non-negotiables (aesthetic match + §9 alignment) and stays comfortable in the SVG zone for v1's projected vault scale (40 → 200 nodes). Bundle size advantage is a free side-effect.

## The question

§10.18 (per `RIFT_V2_VISION.md` lines 338-349) held five sub-decisions until Phase 8 begins:

1. Rendering tech (Canvas / WebGL / native)
2. Graph layout algorithm (force-directed / manual / hybrid)
3. Node type taxonomy (refinements beyond "Index vaults + integration enrichments")
4. Performance at vault scale
5. State persistence (zoom, pan, opened files, layout)

Items 3 (taxonomy) and 5 (persistence schema) are independent of library choice — both fall out the same way regardless of who renders. Items 1, 2, and 4 are the lib-shaped questions; this doc resolves them. Taxonomy refinement is deferred to Phase 8 implementation; persistence schema lands as part of `rift-config.toml` extension during Phase 8.

The Phase 8 description in `RIFT_V2_PHASE_PLAN.md` line 121 specifically called for "render same fixture in all 3 libs, decide on perf + interaction quality." This research beat replaces that spike with documentation-driven evaluation against published API surfaces — the unknowns turned out to be answerable from docs without standing up three parallel POCs. Doc-driven beats spike when the differentiators are *capability-shaped* (what each lib's API allows) rather than *runtime-shaped* (how each lib feels at the same fixture). For Rift, the differentiators are capability-shaped.

## Candidates compared

### A. Rendering tech

| Lib       | Default      | High-perf path | Custom render hook |
|-----------|--------------|----------------|--------------------|
| D3        | SVG (DOM-backed) | Canvas (~200 LOC custom render loop) or PixiJS layer | full primitive control |
| Cytoscape | Canvas       | built-in WebGL extension (instanced drawing) | CSS-like stylesheet, no per-frame callback |
| Sigma     | WebGL only   | (already WebGL)| custom GLSL node/edge programs |

**For Rift's brand:** D3 + SVG is the natural shape. `<feGaussianBlur>` + `<feDropShadow>` filters give pixel-perfect amber-glow + matte-black without library opinions to fight. Sigma's WebGL approach gets the same visual outcome but at the cost of writing fragment-shader GLSL. Cytoscape's CSS-like API doesn't have a glow primitive at all — workarounds via `border-color` + `border-opacity` tricks land at ~80% visual fidelity.

### B. Layout

| Lib       | Force-directed | Pinned-node | Mix pin + sim | Notes |
|-----------|----------------|-------------|---------------|-------|
| D3        | `d3-force` (manyBody, link, center, collide, custom) | `node.fx`/`node.fy` | yes — pin via fx/fy, unpin via null | iterative simulation, settles via alphaDecay |
| Cytoscape | `cose`, `fcose`, `cise`, etc. | `locked: true` on node | yes via `fcose` | Compound Spring Embedder family |
| Sigma     | `graphology-layout-forceatlas2` | FA2 `isNodeFixed` predicate | yes | Web Worker supervisor available |

All three handle the layout requirements. D3 + d3-force is the most flexible for custom forces (e.g. category-cluster gravity wells, boundary containment for the cockpit-panel viewport). Cytoscape's fcose is the easiest out-of-box. Sigma's FA2 with the worker supervisor is best for very large graphs but adds an off-thread architecture concern.

### C. Performance at vault scale

Today's Abyssal Index = ~40 vaults. Phase 8 v1 estimate = 50-200 nodes (vaults + integration-enrichment nodes). Long-term growth is open-ended but bounded by human curation effort.

- **D3 SVG** comfortable to ~500 nodes at 60 FPS (community consensus; D3 docs publish no benchmarks). Cliffs at 1000+. Canvas escape hatch is documented (~200 LOC) but adds custom hit-detection work.
- **Cytoscape** Canvas comfortable at ~500-1000; WebGL extension scales to 10K+ via instanced drawing (verified via Context7).
- **Sigma** WebGL handles 5K+ in their own benchmarks (`large-graphs` story confirmed); FA2 layout converges in seconds at 5K, instant at our scale.

For v1's 200-node target, all three have headroom. For v2+ growth past 500 nodes, Cytoscape and Sigma have less work; D3 needs the Canvas hybrid. This is a real cost — but not a v1 cost, and the perf-cliff trigger is well-documented in the codebase by the time it matters.

### D. Real-time updates (§10.15 Channel<T> fit)

All three handle incremental add/remove/update with batching:
- **D3** — enter/update/exit via `selection.join(enter, update, exit)` with key function `.data(nodes, d => d.id)` for stable identity. Restart simulation via `simulation.nodes(newArray).restart()` without full re-layout.
- **Cytoscape** — `cy.batch()` / `cy.startBatch()/endBatch()` wraps multiple ops in single reflow. `cy.add()` / `cy.remove()` for incremental.
- **Sigma** — `graphology` mutations (addNode/dropNode/setNodeAttribute) + `renderer.scheduleRefresh()` to defer rendering to next frame.

D3's key-function discipline aligns best with envelope-stream identity propagation (envelopes carry stable IDs; D3's data-join uses them directly).

### E. Drag-into-terminal (§11 friction-reduction primitive)

All three require manual HTML5 drag-bridging — none of them ship a "drag node out of graph into another DOM element" primitive. Cleanest plumbing:
- **Sigma** — `downNode` event payload + `viewportToGraph()` exposes coordinates and node ID; trivial to start an HTML5 `dragstart` from there.
- **D3** — `simulation.find(x, y)` for node hit-detection + custom drag handler that defers to native `dragstart`. Most natural fit because D3 is DOM-native and xterm.js is DOM-native — no canvas/WebGL boundary to cross.
- **Cytoscape** — moderate plumbing; bridge from Cytoscape's `tap`/`drag` events into HTML5 drag-and-drop. Most lines of glue code of the three.

## Decision: D3 (modular)

**Stack pin (Phase 8 entry — additive only, no version-substitution drift):**

```jsonc
// package.json devDependencies (frontend)
{
  "d3-force": "^3.0.0",       // simulation engine
  "d3-selection": "^3.0.0",   // DOM binding + data join
  "d3-zoom": "^3.0.0",        // pan/zoom transform
  "d3-drag": "^3.0.0",        // drag behavior
  "@types/d3-force":     "^3.0.10",
  "@types/d3-selection": "^3.0.11",
  "@types/d3-zoom":      "^3.0.8",
  "@types/d3-drag":      "^3.0.7"
}
```

**Why modular not full `d3`:** Full `d3` is ~200 KB gzipped and includes geo, time-format, contour, etc. — none relevant to the Index graph view. Modular install is ~50 KB and survives tree-shaking cleanly.

**No `d3-array`, `d3-scale`, `d3-shape` initially.** Add only when actually needed (color scales, axis ticks, etc.). Each addition is a follow-up decision; do not pre-emptively pin.

## Why D3

Three reasons, ranked by weight:

1. **Aesthetic match is exact and free.** Rift's brand (matte black + amber/terminal lanes + scanlines + CRT vignette per §10.3) is non-negotiable. SVG filters give pixel-perfect amber glow via `<feDropShadow flood-color="#f59e0b">` with no library opinions to fight. Cytoscape requires CSS-tricks workarounds (border + opacity manipulation); Sigma requires custom GLSL fragment shaders (~3h of shader work). D3 + SVG is the cheapest path to brand fidelity by a wide margin.

2. **§9 Integration Decoupling alignment.** D3's primitives-first stance maps cleanest to Rift's decoupled architecture — frontend ships a render layer, not an opinionated graph viewer. Cytoscape and Sigma are higher-level libs with baked-in worldview (e.g. Cytoscape's stylesheet selectors, Sigma's program registry). Rift's frontend already chose primitives over frameworks elsewhere (Svelte 5 runes, vanilla xterm.js mount); D3 continues the pattern.

3. **Vault scale stays in the comfortable SVG zone for v1.** 40 → 200 nodes is well inside D3-SVG's 60 FPS envelope. The Canvas escape hatch (~200 LOC) is documented and reachable post-v1 if Index growth blows past 500. WebGL via PixiJS layer is a v2 question, not a v1 blocker.

Bundle size (~50 KB modular vs Cytoscape's heavier footprint) is a free fourth reason but not load-bearing — Tauri webview handles all three fine.

---

## Why not Cytoscape

Cytoscape is the lowest-effort path if brand fidelity is *negotiable*. It isn't — §10.3's aesthetic match is a §7 "fired list" anti-pattern guard ("No shortcuts on load-bearing visuals"). The CSS-like stylesheet API doesn't have a glow primitive; the workaround (border-width + border-opacity tricks) lands at ~80% visual fidelity, which fails the "match the mockup, do not improvise" rule.

Cytoscape would be the right call if Index were expected to grow past 1000 nodes during v1 (its WebGL extension scales effortlessly), or if engineering hours mattered more than visual polish on this surface. Neither is true for Rift v1.

Recorded for posterity in case the constraints flip post-v1: Cytoscape v3.33.2, MIT, zero deps, fcose extension for force + pinning, `cy.json()` for state persistence, batched updates via `cy.batch()`. Maturity is excellent (released 21 days before this decision).

---

## Why not Sigma

Two strikes:

1. **WebGL-only with no Canvas fallback.** Sigma throws hard if WebGL context unavailable or lost. Tauri 2's webview is fine on modern Windows desktops, but the §13 post-v1 native-Android/SSH-headless work would need a fallback path or a graceful error UI. D3-SVG is portable to every webview Rift might ever ship in.

2. **Glow effect requires custom GLSL.** Aesthetic compliance estimated at ~3-4h of shader work to get the amber-glow + matte-black exactly right. D3's SVG filters get there in minutes.

Sigma would be the right call if vault scale exploded past 5K nodes during v1 (its WebGL renderer + FA2 worker is the best of the three at that scale) and if the team had GLSL comfort. Neither applies.

Recorded: Sigma v3, MIT, ~150-200 KB gzipped (Sigma + graphology + graphology-layout-forceatlas2), `getCamera().getState()` for camera persistence, `graph.export()` for full graph state.

## Risks + mitigations

### Risk 1: SVG perf cliff at ~500 nodes

**Likelihood:** medium-low for v1 (200-node target leaves headroom). Higher post-v1 if Index curation accelerates.
**Impact:** rendering janks at 1000+ nodes; force simulation drag becomes choppy.
**Mitigation:** Canvas hybrid escape hatch (~200 LOC custom render loop fed by d3-force position output). Trigger condition: 60 FPS drops below 30 sustained at user-typical zoom level with default vault count. Document the trigger in `r004` so the decision to migrate is mechanical, not judgment-based.

### Risk 2: Svelte 5 `$effect` lifecycle subtlety

**Likelihood:** high during initial Phase 8 implementation; common D3+component-framework gotcha.
**Impact:** memory leaks (simulation tick listener not detached on unmount), or D3 selections fighting Svelte's hydration (selections run before mount = ghosting).
**Mitigation:** Document the canonical mount pattern in `r004`:

```svelte
<script lang="ts">
  import { onMount } from 'svelte';
  import * as d3 from 'd3-force';

  let container: HTMLElement;
  let simulation: d3.Simulation<...>;

  $effect(() => {
    if (!container) return;
    simulation = d3.forceSimulation(nodes)
      .force('link', d3.forceLink(links).id(d => d.id))
      .force('charge', d3.forceManyBody().strength(-300))
      .on('tick', tick);
    return () => {
      simulation.stop();
      simulation.on('tick', null);
    };
  });
</script>
```

### Risk 3: d3-force `alphaDecay` vs `alphaTarget` confusion

**Likelihood:** medium during tuning.
**Impact:** simulation never settles, or settles too fast and looks lifeless.
**Mitigation:** Pin defaults in code comment + `r004` entry: `alphaDecay: 0.0228` (default, fine), `alphaTarget: 0` (target alpha when at rest), `alpha(0.3).restart()` for graph-mutation-driven re-heating. Capture the rule once; future tuning is a knob-twist, not a re-derivation.

### Risk 4: Drag-into-terminal HTML5 bridge complexity

**Likelihood:** medium-low (D3 is DOM-native; xterm.js is DOM-native; no canvas/WebGL boundary to cross).
**Impact:** drag-from-graph drops onto terminal but content arrives garbled or doesn't trigger insertion.
**Mitigation:** Reuse Phase 6.6 drag-node-into-terminal primitive (commit 3b91fbb) — that infrastructure already exists for the file-tree node case; the graph-node case is the same shape with a different source DOM element. Phase 8 implementation should grep for `dragstart` handlers in `src/lib/Cockpit.svelte` and extend rather than reinvent.

### Risk 5: `vault-staleness-on-implementation-substitution`

**Likelihood:** present whenever a stack decision is made and not enforced.
**Impact:** Phase 8 implementation silently picks a different lib because the locked decision wasn't load-bearing in CI.
**Mitigation:** This decision doc is the load-bearing artifact. `tools/check-translator-boundary.sh` does NOT need extending (graph lib is frontend, not Rust translator surface). If a future Phase 8 commit drops `cytoscape` or `sigma` into `package.json`, the diff alone surfaces the deviation; no automated guard required for v1.

## What v1 explicitly does NOT do

- **No multi-graph or graph-of-graphs.** Single Index graph view. The "graph of project graphs" idea (each project as a sub-graph zoom level) is out of scope for v1; surfaces in v2 only if operational signal demands it.
- **No graph editor / vault-creation-from-graph UI.** The Index graph is a *view* of vault data; vault creation happens via the existing `/vault-update` skill outside Rift. Rift surfaces graph nodes; doesn't author them.
- **No cross-vault link inference.** The graph renders explicit links present in vault frontmatter / index files. Heuristic link discovery (e.g. "these two vaults mention the same pattern, show a faint link") is post-v1.
- **No layout-spec persistence beyond camera + node positions.** Force-directed parameters (charge, link distance, etc.) are hard-coded in v1; per-user tuning is post-v1 if asked for.
- **No 3D / VR / immersive view.** Always 2D. (Yes, this is being said out loud.)
- **No subscript filter UI in graph view.** Filter UI lives in the Index *tab* (per §10.12 two-views-same-data); the graph reflects whatever the tab filter currently shows.

## v2+ considerations (recorded, not planned)

- **Canvas hybrid via PixiJS** if vault scale grows past 500 nodes sustained. ~200 LOC custom render loop fed by d3-force; preserves D3 idiom, preserves SVG filters as a fallback layer for "selected node" overlay.
- **WebGL via Sigma migration** if Canvas hybrid proves insufficient. This would be a real lib swap, with pr003 entry capturing the rationale. Not a v1 concern.
- **Custom force functions** for category clustering (e.g. `pr*` vaults gravitate toward one corner, `r*` toward another). Cosmetic enhancement; v1 lets force-directed find its own equilibrium.
- **Saved layouts** — user pins, the layout snapshot persists across sessions. v1 stores positions; v1.x adds a "save/load layout" menu if asked for.
